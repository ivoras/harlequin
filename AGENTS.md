# Harlequin — Agent Guide

Client-server AI agent system in Go. A REST + WebSocket **server** talks to LLMs, stores data in SQLite, runs an agentic tool-calling loop, and manages skills; a Bubble Tea **TUI client** and a Svelte **web UI** talk to it. Multi-user, per-organisation.

## Binaries
- `cmd/harlequin-server` — REST + WebSocket API, LLM access, SQLite, agent loop, skills.
- `cmd/harlequin` — TUI client (Claude Code-like; dark-purple/light-green theme).
- `web/` — mobile-first browser SPA (Svelte 5 + Vite; not a Go binary). `make web-build` → `web/dist`, served by the server when `server.web.dir` is set. Run `make web-build` after editing `web/`. User-facing features go in **both** clients (TUI + web).

## Stack
- Go 1.25. Router `go-chi/chi/v5`; Agent session streams, and chat are over WebSocket (`coder/websocket`).
- SQLite via `mattn/go-sqlite3` (CGO, `-tags sqlite_fts5`) + `asg017/sqlite-vec` (`vec0`). Build: `CGO_ENABLED=1 go build -tags sqlite_fts5 ./...` (or `make build`). Compile-time headers are vendored in `third_party/sqlite/`; no system `libsqlite3-dev` needed.
- TUI: `charm.land/{bubbletea,bubbles,lipgloss}/v2`, `glamour` + `chroma` for rendering.
- JS engine: `dop251/goja` (ES5.1-compatible, much of ES6 incl. let/const, classes, Promise, BigInt; sandboxed) for `<?js ?>` skill templating and the `run_js` tool.
- Config: YAML (structure) + `.env` (secrets); env overrides YAML.

## Layout
- `internal/server/{config,db,storage,auth,api,llm,embed,jsrun,agent,memory,documents,conversation,dom,sandboxfs,pdfextract,mdtmpl,audit,sessionlog,usage,session,sessionhub,presence,project,projectchat,skills,userconfig,cron,mcp,webfetch,email,notify,notifyx,telegram,secrets}` — `session` is the chat-session store (the `sessions` table + messages); `sessionhub` is the live-session runtime (one goroutine per active session; see Live sessions below); `project` is the project registry + chatroom store and `projectchat` the chatroom broadcast hub (see Projects below); `notify` is the notification store and `notifyx` the delivery dispatcher (in-app / email / Telegram).
- `internal/server/skills/jstmpl` — PHP-style `<?js ?>` templating (on `jsrun`).
- `internal/client/{config,apiclient,tui,skills}`
- `internal/shared/types` — REST DTOs. `internal/server/db/migrations/{system,shared,user,project}/*.sql` and `skills/` embedded via `embed.FS`.

## Concepts
- Layered memory + documents: user + shared (organization-level) + projects
- Skills are a set of virtual documents: the main skill file, referenced as "skill://name", and optional files such as scripts, referenced e.g. as "skill://name/file/name.md"
- All search (docs, memory, skills) fuses project scope (if active), then shared scope, then user scope
- A **hat** is a set of specialised skills that **overlay and take precedence over** whatever the project/shared/user scopes would resolve: while a hat is worn, its skill copies (files under `skills/<name>/...` in the hat) win outright, and its optional frontmatter `skills:` list restricts which skills are visible. The point is that certain people need their specialised skill variants for certain kinds of tasks, without touching the originals. A hat may also replace the system prompt (its `system_prompt.md` body). A user wears at most one hat per session; hats and projects are independent of each other. Managed by owners/admins: TUI `/hat create|edit|files|addskill|rmskill|del`, web Hats view; REST `POST /hats`, `GET|PUT /hats/{name}/files/*`, `POST|DELETE /hats/{name}/skills[/{skill}]`.

## Projects (shared workspaces)
- A **project** is a shared workspace grouping members, sessions, documents, and memories, plus a live chatroom. Registry (projects/members/invites) is in the **system** db (`migrations/system/0002_projects.sql`, `internal/server/project`). Only one project can be active. Any user can create a project (becoming the first member); any member can invite others (by email; the invitee gets a notification and accepts) and **assign** a session to the currently active project.
- **Assigning** a session *moves* it (and its messages) from the owner's `user.db` into the project db (`project.MoveSessionToProject`), so a session belongs to at most one project and every project member sees it. Project sessions are a **shared live session**: the sessionhub keys them by `p/<projectID>/<sessionID>` (vs `u/<userID>/…`), so all members attach to the same live turn; `agent.Run` takes a `projectID` and runs the turn on the project db (`rc.sessDB`) while user-scoped tools (skills/MCP/cron/usage) stay on the acting user's db. WS: `GET /projects/{id}/sessions/{sid}/ws`.
- **Project memory** uses composite-id scope `p.<localid>` in the project db; a project session fuses user+shared+project (`memory.SearchFused`) and `memory_write`/auto-extraction default to the project scope (any member may write it).
- **Chatroom**: `internal/server/projectchat` (broadcast hub, no turn buffer), communication over the websocket; messages persist in the project db (`chat_messages`). Clients show a side-pane only while a project is active (web right-hand pane / TUI split; `/project` to manage, `/say` in the TUI to post). Hats stay orthogonal to projects.
- **Documents are scoped like memory** (personal/shared/project). Each corpus lives in its tier's db (`user.db` personal, `shared.db` org, `project.db` project) with its own `documents`/`doc_chunks`/`doc_chunks_fts`/`doc_chunks_vec`. `documents.SearchScoped`/`ScopesFor(userDB, projDB)` fuse the corpora with the same RRF as a single store — a project session adds the project corpus (`rc.projectDB` is set only then), otherwise personal+shared — and every hit is **scope-labelled** (`SearchResult.Scope` ∈ personal/shared/project; chunk ids are `d.{u,s,p}.<n>`). `search_docs` (agent), `GET /documents/search` (`?project=<id>` to include a project), and the skill `SearchDocs` helper all fuse; ingest picks the target corpus via `CreateDocumentRequest.Scope` (`shared` default, `personal`, or `project`+`project_id`).

## Storage
- `internal/server/storage.Manager` owns the handles. `data/harlequin.db` (system: contains users, api_tokens) and `data/shared.db` (org: shared memories, documents, org-scope skills incl. the seeded baked-ins) are **kept open**. 
- Each user has `data/users/<id>/user.db` (their memories, sessions, messages, usage, audit, user-scope skills, cron jobs, and a generic `config` key/value table), **opened and closed per request** via `WithUser`; `EachUser` fans out for admin aggregation and maintenance sweeps. Uploaded files go to `data/shared_files/` and `data/users/<id>/files/`.
- Each project's own data lives in `data/projects/<id>/project.db` (`db.Project` role; `storage.WithProject`/`EachProject`), holding project-scope memories, docs, skills, mirroring the per-user lifecycle. 
- Hats are org-wide and live in `shared.db` (`hats`/`hat_files`; per-hat skill overrides are `hat_files` rows under `skills/<skill>/...`). Baked-in hats are seeded like skills; any legacy `<data_dir>/hats/` directory is imported once at startup. Every user can activate a hat, only admins and owners can edit them (`PUT|DELETE /hats/{name}`).
- No cross-file foreign keys: memory ids are composite strings `u.<localid>`/`s.<localid>`/`p.<localid>`, and `memory_conflicts` endpoints/ids are composite (conflicts involving a user memory live in that user's db; shared–shared in `shared.db`).

## Core flows
- **Auth**: email/password login -> issued API bearer token (SHA-256 in DB); middleware injects user/role. Email is the identity (`users.email`). Self-registration (`/auth/register` -> emailed 6-digit magic code -> `/auth/verify`) holds signups in `pending_registrations` until verified; gated by `auth.allow_registration` (default true). Codes go through `internal/server/email` (SMTP, or logged to console when `smtp_host` is empty).
- **Roles** (`types.IsOwner`/`types.IsElevated`): `owner` > `admin` > `user`. Owner is the only role that can create/edit users (`requireOwner`). Owner+admin ("elevated", `requireElevated`) may create/delete shared memories, delete documents, read the audit log, publish skills, and view other users' usage. Ordinary users get their own sessions and user-scoped memories only — the `memory_write` tool refuses `shared` scope for them.
- **Chat**: client connects the session WebSocket (`GET /sessions/{id}/ws`) and submits a prompt frame -> agent loop composes prompt with resolved skills, calls the LLM provider (routing + fallback), dispatches tool calls (`memory_search`, `memory_write`, `memory_change`, `memory_delete`, `memory_feedback`, `ask_user`, `notify_channels`, `calculator`, `list_skills`, `load_skill`, `run_js`, `search_docs`, `Grep`, `cron_create`/`cron_list`/`cron_update`/`cron_run`/`cron_delete`, `WebFetch`, `WebFetchDOM`, `WebFetchGrep`, skill-defined tools, MCP tools), loops to `max_steps`, streams event frames. `memory_write`/`memory_change` surface detected conflicts in their result so the model can warn the user; `ask_user` emits an `ask_user` event and ends the turn so the user can reply.
- **Live sessions** (`internal/server/sessionhub`): each active session is a server-side goroutine that runs turns under a *session-scoped* context (over WebSocket), so a client disconnect does not cancel the turn. It buffers the in-flight turn's events with a session-monotonic `Seq` and fans them out to all connected WebSocket subscribers; the goroutine exits after `sessions.idle_timeout` (default 30m) with no connection and no running turn. **Resume**: a (re)connecting client sends `hello{have_seq}`; the server replies `synced{running, committed_through, seq}` then replays buffered events with `seq > have_seq` and continues live. Cold resume (have_seq 0): the client loads committed messages via `GET /sessions/{id}/messages`, renders those with `id <= committed_through`, and reconstructs the in-flight turn from the replayed buffer (which re-emits it from a `user_message` echo). WebSocket auth accepts the bearer token from the Authorization header (Go clients) or the `bearer.<token>` subprotocol (browsers). The Web UI keeps the session id in `?c=<id>` so a refresh resumes; the TUI resumes via `/resume` (a session picker), `/resume <id>`, or the `--session <id>` launch flag.
- **Notifications** are **pushed** over the session WebSocket, not polled: each connection runs a server-side poller (`Server.pollNotifications`, ~5s) that reads the user's pending notifications (the DB is the store, `notify.ListPending` filtered by interface) and sends new ones as `notification` events; clients ack over REST (`POST /notifications/{id}/ack`). Clients render passive notifications in a **persistent alert box** above the transcript (not in the session transcript), kept until dismissed (web ✕/Run buttons; TUI `/dismiss [n|all]` and `/run <n>`); session-title updates and auto-run prompts are handled as control/behaviour, not shown as alerts. The same notifications are the intended payload for future delivery to other interfaces (e.g. Telegram).
- **Skills**: live in the scoped databases; resolution precedence **project -> shared -> user** (deeper scope wins, same order as docs/memory). Baked-in skills **and hats** are embedded in the binary and **seeded into `shared.db`** on startup (`skills.Seed`, hash-guarded via the `*_seed_hashes` tables: human edits in the DB are preserved). Top-level baked files (e.g. `skills/system_prompt.md`) are not skills and are read straight from the binary. Server is the single source of truth; clients pull/push via `/skill` slash-commands. Skills support inline `<?js ?>` and frontmatter-declared tools.
- **WebFetch** (`internal/server/webfetch`, on by default, `agent.web_fetch.enabled: false` to disable): fetches a URL with browser-like anti-bot measures — realistic Chrome headers, a uTLS Chrome JA3/TLS fingerprint, HTTP/2, a cookie jar, redirect following, request jitter (no headless browser, no CAPTCHA solving) — converts HTML to Markdown (`JohannesKaufmann/html-to-markdown`), then analyses it with a small model (`agent.web_fetch.model`) given only the WebFetch tool. http→https upgrade; 15-minute per-URL cache; SSRF guard blocks private/loopback targets unless `allow_private`. **Zyte fallback** (when `ZYTE_API_KEY` is set): a direct fetch that returns HTTP 4xx is retried through Zyte's `browserHtml` API, and the URL's domain is remembered in a thread-safe in-memory set so later fetches of that domain skip straight to Zyte (`via_zyte` in the log). The fallback lives in `webfetch.Client.FetchRaw`, so it covers every agent-initiated fetch alike — the WebFetch tool, the sandbox `fetch()` used by JS cron jobs, and the web-monitor skill (one shared `Client`). The trajectory logs each fetch (`web_fetch`: url, final_url, cached, fetch_ms, bytes) and the delegated inner LLM call (`delegated_llm_request`/`delegated_llm_response`, labeled `delegate: web_fetch`, with `duration_ms`). Companion tools on the same fetch pipeline: **WebFetchDOM** (sandboxed JS over the parsed DOM, `internal/server/dom` + `sandboxfs`) and **WebFetchGrep** (deterministic regex over the fetched HTML; an optional prompt runs the analysis model over the matches).
- **Session logging**: full chat trajectory written as JSONL at `<data_dir>/sessions/<user_id>.<conv_id>.jsonl` (ids zero-padded to ≥5 digits). Optional via `sessions.enabled` (default true).
- **Interfaces**: each session is tied to one *interface* (the medium a user talks through) plus its *API* (transport), stored on the session (`api`, `interface` columns) and logged in `session_start`. REST clients announce the interface via the `X-Harlequin-Interface` header (TUI → API `REST` / interface `TUI`; the web SPA → `Web`); cron-started sessions use `Cron`/`Cron`; a planned Telegram chat bridge sets `Telegram`/`Telegram` (today Telegram is outbound-only: `notifyx` delivers notifications via `internal/server/telegram`). Add an interface by announcing a new header value, or set api/interface directly in a server-side bridge. Constants in `internal/shared/types` (`InterfaceTUI`, `APIREST`, …).
- **Per-user config**: generic key/value `config` table in `user.db`, via `internal/server/userconfig.Store` (Get/Set/Delete/All) and REST `/config`, `PUT|DELETE /config/{key}` (TUI `/config`). Holds small settings that don't warrant their own table — e.g. registering a Telegram connection (`telegram.chat_id`, `telegram.username`).

## Conventions
- **"Simple things should be easy, complex things should be possible."** The zero-config path must just work — sensible defaults everywhere (scope, model suite, session handling), no flag or parameter required for the common case. But the advanced case must have an escape hatch rather than a wall: explicit scopes, qualified references (`p<id>.N`), `all_projects` search, per-tool options, YAML + env overrides. When adding a feature, design the one-liner first, then make sure power users can reach the rest — never the reverse.
- **Principle of least astonishment (POLA) governs every user-facing decision** — features, defaults, tool behavior, references, errors. Harlequin is a multi-user assistant whose users can't read the code: whatever a reasonable user would predict (from names, conventions, or what a sibling feature does) is the spec. Concretely: safe defaults (personal scope, not shared), symmetric capabilities (anything visible on the read side must be writable/openable where the user would expect), references that mean the same thing everywhere they appear, and failures that are loud and persistent rather than silent or transient.
- Comments should be terse and explain the non-obvious (the *why*, edge cases, gotchas); don't restate what the code plainly says.
- Secrets only in `.env`, never in YAML or code.
- URLs built with names in the path (skill name, hat name, file relpaths, config keys, …) must URL-encode those strings: Go `url.PathEscape` per segment (`apiclient.escapePathSegments` for relpaths), web `encodeURIComponent` (the `q()` helper in `web/src/lib/api.ts`).
- `docs/user_guide.md` is the user-facing documentation. Its "Slash commands in the TUI" chapter must be kept in sync: whenever a TUI slash command is added, removed, or changes syntax (`internal/client/tui/slash.go`, `helpText` in `commands.go`), update that chapter in the same change.
- All JS (templating, `run_js`, skill tools) runs through the sandboxed `jsrun` runner: no file/network (except allow-listed `fetch`), hard `vm.Interrupt` timeout, output cap.
- **Autonomous LLM judgments**: any server process that is *not* directly started by the user (background workers, post-turn hooks, scheduled jobs, etc.) and asks the LLM to *judge* whether to take an action must:
  1. **Request a confidence score** — instruct the model to attach an integer `confidence` field (1–10) to every judgment, meaning how sure it is the result is correct and safe to act on without user confirmation.
  2. **Gate on the score** — only accept and act on judgments with `confidence >= 7`. Discard or ignore everything below the threshold.
  3. **Prefer structured output** — use JSON (or an equally machine-parseable format), not free-form prose lines.
  4. **Silence on uncertainty** — if nothing meets the bar, return an empty collection or explicit negative; never persist placeholder/meta text (e.g. "no facts found").
  - Shared helpers live in `internal/server/llm/judge` (`MinConfidence`, `Accept`, `Clamp`, `PromptRules`, `ParseJSONObject`). Domain-specific parsers (e.g. `internal/server/agent/memextract`) embed `judge.PromptRules()` and call `judge.Accept` before side effects.
- Plan of record: `.cursor/plans/harlequin_agent_base_*.plan.md`.
- When coding, ask the user for input (interview them) if you notice any inconsistencies or ideas for improvement.
