# RAG embedding research — agent guide & decision record

This folder holds an empirical study that selects a single
`embedder / chunker / lexical_backend` for Harlequin's document ingestion (and
memory) retrieval. The current report is **`paper3.html`** (generated by
`make_paper3.py`). Earlier rounds (v1 `paper.html`, v2 `paper2.html`) were
removed; only v3 and the shared inputs remain.

This file records the decisions that steer the report so a future agent can
extend it without re-litigating them. **Change the report only in ways
consistent with these decisions, or update this file when a decision changes.**

## Goal

Pick ONE embedding model for Harlequin, plus the chunker and lexical backend to
pair it with. Optimised for document ingestion, but memory-slot lookups and
out-of-domain rejection are first-class concerns too.

## Decisions that steer the study

1. **Single model the whole path.** Each candidate is served *alone* on
   `:2235` (llama.cpp, OpenAI API) and does everything — per-sentence boundary
   embeddings, chunk vectors, and query vectors. No result mixes models. Manage
   the server with `r3_server.py` (one model resident at a time).
2. **Five candidates** (all in `~/LLM`): granite-311M, snowflake-arctic-l-v2.0,
   Qwen3-Embedding-0.6B, embeddinggemma-300M, LFM2.5-Embedding-350M.
3. **Same corpus + questions as before.** TEU PDF → `data/corpus.json`; reuse
   `data/eval_questions.json` (802 in-doc incl. 122 misspelled / 100 "dumb",
   201 OOD) and `data/answer_sets.json`. Generalization reuses the Darwin
   corpus (`data/corpus2.json` + `data/questions2.json`).
4. **Similarity = cosine** everywhere (sqlite-vec `vec0` cosine; normalised
   cosine in the generalization brute-force path).
5. **Metrics** are duplicate-aware (a hit covers any acceptable sentence),
   size-normalised (`answer@Btok` = hit within a B-token retrieved budget, to
   defeat the chunk-size confound), with 95% bootstrap CIs, plus OOD AUC/EER.
6. **Prompt modes: evaluate BOTH and report the gap.** Each model is run with
   its native prompt convention and with no prefix (raw text = how Harlequin
   embedded before). The gap is measured on ONE reference chunker
   (`mech_512`, dense); the better mode per model is *carried* into every later
   chapter (`MODES_OF` / `carried` in `r3_aggregate.py`). Native query/doc
   prefixes live in `lib.EMBEDDERS`. Finding: only snowflake's `query:` prefix
   helps; qwen/gemma/lfm2 do best on raw text.
7. **Per-model semantic gate.** `sem_adjacent` cuts where adjacent-sentence
   cosine distance exceeds a gate. The gate's scale is model-specific, so it is
   swept at percentiles of THAT model's adjacent-drift distribution
   (`GATE_PCTILES` in `r3_build.py`), and the best gate is chosen by the same
   composite that ranks models.
8. **Chunkers compared (8):** `per_sentence`, `mech_256/512/1024/1500`,
   `overlap_512/1024`, and the model's best `sem_adjacent`. (v2's `structure_*`
   was dropped — it is corpus-structure-specific.)
9. **Lexical backends:** custom numpy Okapi `bm25` vs SQLite `fts5`, each
   standalone and as a dense+lexical RRF hybrid. Naming is
   `embedder/chunker/lexical_backend` with lexical ∈
   {`dense`,`bm25`,`fts5`,`bm25_hybrid`,`fts5_hybrid`}. `bm25_hybrid` is the
   dense+numpy-BM25 RRF (named symmetrically with `fts5_hybrid`).
10. **Winner = pinpoint+rejection composite:** weighted mean of z-scored
    metrics, `2×`{R@1, MRR, AUC} + `1×`{R@5, answer@1024} (`COMPOSITE_W`). Each
    model is represented by its own best `(chunker, lexical)`; models are ranked
    by the same composite.
11. **Cost is reported, not scored** (quality-first): dim (vec0 storage + search
    cost) and embed throughput (`data/r3_bench.json`) are tabulated as a
    tie-breaker the reader can override.
12. **Do NOT re-introduce v2 dead ends:** HyDE, contextual-prefix embedding, and
    zero-shot LLM reranking did not help (rerank hurt) and are excluded.
13. **Report style:** terse; every table is sortable and every column header
    carries a plain-language `title=` tooltip (`TIPS` in `make_paper3.py`).
14. **RRF weight chapter (§8):** sweep the FTS5-arm RRF weight on the winning
    `…/fts5_hybrid` (dense fixed at 1.0), 5 values, scoring misspelled vs clean
    subsets separately (`r3_rrf_sweep.py`). Lexical weight helps clean queries a
    little and hurts misspelled ones a lot. **Caveat (do not overclaim):** the
    question set is ~99% paraphrase — only 8/802 queries carry an exact
    numeric/quoted token — so it under-samples the exact-extraction case FTS5
    exists for (article numbers, terms of art). The result means "don't
    *over*-weight FTS5", not "drop it"; keep the arm (weight > 0) as recall
    insurance.
15. **Exact-extraction probe (§8.1).** `r3_probe_build.py` mines the corpus for
    grounded queries anchored on exact tokens (percentages, protocol numbers,
    article+paragraph refs) → `data/probe_exact.json`; `r3_probe_sweep.py` (needs
    the snowflake server) runs dense / FTS5-weight / **score-gated** hybrids on
    the probe vs the paraphrase set → `data/r3_probe_sweep.json`. Finding: the
    weight preference inverts (exact wants FTS5 up-weighted, dense alone weakest);
    and **dropping low-BM25 FTS5 hits before fusion (score-gating) beats a flat
    weight on both query types** — the recommended production fusion is a
    confidence-gated (ideally gated + up-weighted) lexical arm, not a fixed
    global weight.

## Gotchas (learned the hard way)

- **No concurrent embedding.** This llama.cpp build aborts in
  `server_context decode` (ggml_abort) if >1 `/v1/embeddings` request is in
  flight. `lib.Embedder` defaults to `workers=1`; do not raise it.
- **Token cache is per-model.** Chunk budgets use the model's own tokenizer via
  `/tokenize`; `lib.gtok` is namespaced by `lib.GTOK_MODEL` (set per build).
- **Doc-vector reuse.** Configs whose doc prefix is empty share doc vectors via
  the on-disk cache (`data/embed_cache.sqlite`), so `snowflake_np`/`qwen06b_np`
  reuse the native index and only re-embed queries; gemma/lfm2 have distinct
  doc prefixes so they build separate `_np` indexes.
- **Eval is offline once built.** `r3_eval.py` only needs the server if query
  vectors aren't cached; reruns (e.g. renames) read cached vectors + on-disk
  indexes, no server.

## Pipeline / how to run

```
python3 r3_run.py [models…]   # per model: start server, bench, build, eval, stop
python3 r3_eval.py <cfg>      # re-eval one config offline (cached vectors)
python3 r3_aggregate.py       # -> data/r3_agg.json (prefix gap, gates, composite)
python3 r3_rrf_sweep.py       # -> data/r3_rrf_sweep.json (§8 lexical-weight sweep)
python3 r3_corpus2.py <cfg>   # -> data/r3_corpus2.json (generalization, winner)
python3 make_paper3.py        # -> paper3.html
```

Shared code: `lib.py` (embedder registry, cache, vec0 store, token counts),
`chunkers.py`, `bm25.py`. Indexes in `data/r3idx/{config}/`; per-config metrics
in `data/r3_results_{config}.json`.

## Current result

Winner **`snowflake/sem_adjacent(g≈0.43)/dense`** (R@1 0.626, R@5 0.873,
AUC 0.960). For documents, retrieval is `fts5_hybrid` (dense+FTS5 RRF) but the
strong dense arm wants the FTS5 weight ≤1. Cost runner-up: `gemma_np`
(dim 768, ~2.5× throughput). Snowflake is the only model whose query prefix
helps. Harlequin was reconfigured accordingly (snowflake + `query:` prefix on
queries only; `semadj` document chunker); memory/doc vectors must be re-embedded
(`harlequin-server reindex-vectors`) and `search_max_distance` retuned to
snowflake's similarity scale.
